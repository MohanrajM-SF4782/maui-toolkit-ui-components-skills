## Overview

The OTP Input component supports three distinct input types, each optimized for different authentication scenarios. The `Type` property controls which characters users can enter and how the input is displayed.

**Available Types:**
- `Number` — Numeric-only input (0-9)
- `Text` — Alphanumeric input (a-z, A-Z, 0-9)
- `Password` — Hidden/masked input (displays bullets instead of characters)

The default type is `Number`, suitable for most SMS-based OTP scenarios.

## Number Type

The `Number` type restricts input to numeric characters only (0-9). This is ideal for OTP codes received via SMS or generated by authentication apps.

### When to Use
- SMS-based OTP verification (e.g., "+1-234-567-8900 sent code 123456")
- Phone number confirmation flows
- Time-based one-time passwords (TOTP) from authenticator apps
- Standard 4-6 digit PIN codes

### Implementation

**XAML:**
```xml
<otpInput:SfOtpInput x:Name="NumericOtp"
                     Length="6"
                     Type="Number"
                     StylingMode="Outlined" />
```

**C#:**
```csharp
var otpInput = new SfOtpInput
{
    Length = 6,
    Type = OtpInputType.Number,  // Numeric only
    StylingMode = OtpInputStyle.Outlined,
    Value = "123456"
};
```

### Behavior
- Only numeric characters (0-9) are accepted
- Non-numeric input is ignored (user cannot type letters)
- Keyboard shows numeric keypad on mobile devices
- Leading zeros are preserved (e.g., "001234" is valid)

### Example: SMS Verification

```csharp
public class SmsVerificationPage : ContentPage
{
    public SmsVerificationPage()
    {
        InitializeComponent();
        
        var otpInput = new SfOtpInput
        {
            Length = 4,
            Type = OtpInputType.Number,  // SMS codes are typically 4-6 digits
            StylingMode = OtpInputStyle.Outlined
        };

        otpInput.ValueChanged += async (s, e) =>
        {
            if (e.NewValue?.Length == 4)
            {
                // Auto-verify when complete
                bool isValid = await VerifySmsOtpAsync(e.NewValue);
                if (!isValid)
                {
                    await DisplayAlert("Error", "Invalid code", "OK");
                    otpInput.Value = string.Empty;  // Clear for retry
                }
            }
        };
    }

    private async Task<bool> VerifySmsOtpAsync(string otp)
    {
        // Call authentication service
        return await AuthService.VerifyPhoneOtpAsync(otp);
    }
}
```

## Text Type

The `Text` type allows both letters and numbers in the OTP input. This is suitable for authentication codes that contain alphanumeric characters, such as those sent via email.

### When to Use
- Email-based OTP codes (often longer and may include letters)
- Application-generated codes (more entropy with alphanumeric)
- API access tokens
- Multi-factor authentication codes from email providers
- High-security scenarios requiring longer codes

### Implementation

**XAML:**
```xml
<otpInput:SfOtpInput x:Name="AlphanumericOtp"
                     Length="8"
                     Type="Text"
                     StylingMode="Outlined" />
```

**C#:**
```csharp
var otpInput = new SfOtpInput
{
    Length = 8,
    Type = OtpInputType.Text,  // Alphanumeric
    StylingMode = OtpInputStyle.Outlined,
    Value = "abcd1234"
};
```

### Behavior
- Accepts both uppercase/lowercase letters and numbers
- Non-alphanumeric characters are ignored (spaces, symbols not allowed)
- Input is case-sensitive (a ≠ A)
- Keyboard shows standard text input on mobile devices

### Example: Email Verification

```csharp
public class EmailVerificationPage : ContentPage
{
    public EmailVerificationPage()
    {
        InitializeComponent();
        
        var otpInput = new SfOtpInput
        {
            Length = 8,  // Email codes often longer
            Type = OtpInputType.Text,  // May contain letters
            StylingMode = OtpInputStyle.Outlined
        };

        otpInput.ValueChanged += async (s, e) =>
        {
            // Validate format
            if (IsValidEmailCode(e.NewValue))
            {
                bool isValid = await VerifyEmailOtpAsync(e.NewValue);
                if (!isValid)
                {
                    await DisplayAlert("Error", "Invalid verification code", "OK");
                    otpInput.Value = string.Empty;
                }
            }
        };
    }

    private bool IsValidEmailCode(string code)
    {
        return code != null && code.Length == 8 && code.All(c => char.IsLetterOrDigit(c));
    }

    private async Task<bool> VerifyEmailOtpAsync(string otp)
    {
        return await AuthService.VerifyEmailOtpAsync(otp);
    }
}
```

## Password Type

The `Password` type hides user input by displaying bullets/dots instead of the actual characters. This provides privacy and security when entering sensitive codes in public or over-the-shoulder scenarios.

### When to Use
- High-security OTP entry scenarios
- Public or shared device environments
- Payment authorization codes
- Account access in sensitive contexts
- Any situation where visual privacy is important

### Implementation

**XAML:**
```xml
<otpInput:SfOtpInput x:Name="SecureOtp"
                     Length="6"
                     Type="Password"
                     StylingMode="Outlined" />
```

**C#:**
```csharp
var otpInput = new SfOtpInput
{
    Length = 6,
    Type = OtpInputType.Password,  // Masked display
    StylingMode = OtpInputStyle.Outlined
};
```

### Behavior
- Input characters are displayed as bullets (•) or dots (default)
- The actual value is stored internally and accessible via `Value` property
- Cannot visually confirm entered characters (user must trust their input)
- Underlying input can be numeric or alphanumeric depending on use case

### MaskCharacter Property

The `MaskCharacter` property customizes the character displayed when masking Password type input. By default, a bullet (•) is shown, but you can customize this to use asterisks (*), dots (•), or other characters.

### Setting MaskCharacter

**XAML:**
```xml
<otpInput:SfOtpInput Length="6"
                     Type="Password"
                     MaskCharacter="*" />
```

**C#:**
```csharp
var otpInput = new SfOtpInput
{
    Length = 6,
    Type = OtpInputType.Password,
    MaskCharacter = '*'  // Use asterisk instead of bullet
};
```

### Common MaskCharacter Options

| Character | Visual | Use Case |
|-----------|--------|----------|
| `"*"` | ******* | Traditional password style |
| `"•"` | ••••••• | Modern/native style (default) |
| `"•"` (dot) | ....... | Alternative modern style |
| `"█"` | ███████ | Complete block (maximum privacy) |
| `"#"` | ####### | Distinctive pattern |

### Example: Custom Mask Character

```csharp
// Asterisk mask (traditional)
var otpInput1 = new SfOtpInput
{
    Length = 6,
    Type = OtpInputType.Password,
    MaskCharacter = '*'
};

// Block mask (maximum privacy)
var otpInput2 = new SfOtpInput
{
    Length = 6,
    Type = OtpInputType.Password,
    MaskCharacter = '█'
};

// Hash mask (distinctive)
var otpInput3 = new SfOtpInput
{
    Length = 6,
    Type = OtpInputType.Password,
    MaskCharacter = '#'
};
```

### Complete Password Type Example

```csharp
public class PaymentVerificationPage : ContentPage
{
    public PaymentVerificationPage()
    {
        InitializeComponent();
        
        var otpInput = new SfOtpInput
        {
            Length = 6,
            Type = OtpInputType.Password,
            MaskCharacter = '*',  // Show asterisks
            StylingMode = OtpInputStyle.Outlined,
            Separator = "-"  // With visual separator
        };

        var verifyButton = new Button { Text = "Authorize Payment" };
        verifyButton.Clicked += async (s, e) =>
        {
            if (otpInput.Value?.Length == 6)
            {
                await AuthorizePaymentAsync(otpInput.Value);
            }
            else
            {
                await DisplayAlert("Error", "Please enter complete OTP", "OK");
            }
        };

        Content = new VerticalStackLayout
        {
            Padding = 20,
            Spacing = 20,
            Children =
            {
                new Label { Text = "Confirm Payment with OTP", FontSize = 20 },
                new Label { Text = "(Enter the 6-digit code from your authenticator)", FontSize = 12, TextColor = Colors.Gray },
                otpInput,
                verifyButton
            }
        };
    }

    private async Task AuthorizePaymentAsync(string otp)
    {
        bool authorized = await PaymentService.AuthorizeWithOtpAsync(otp);
        if (authorized)
        {
            await DisplayAlert("Success", "Payment authorized", "OK");
        }
        else
        {
            await DisplayAlert("Error", "Authorization failed", "OK");
        }
    }
}
```

### Security Considerations with MaskCharacter

- **Choose obvious masks** — Use *, •, or █ so users understand masking is active
- **Avoid confusion** — Don't use characters that look like input (avoid "0", "1", "O", "l")
- **Platform consistency** — Choose masks consistent with platform conventions
- **User privacy** — Even with different masks, security is equivalent

### Masking Effectiveness

All mask characters provide the same security level:
- ✅ Visual privacy from observers
- ✅ Protection from shoulder-surfing
- ✅ Masks the actual code value
- ❌ Still vulnerable to keyboard monitoring (use on trusted devices)
- ❌ Not secure on compromised systems

## Choosing the Right Type

| Scenario | Recommended Type | Reason |
|----------|------------------|--------|
| SMS verification (4-6 digits) | **Number** | Expected use case, numeric-only |
| Email verification code | **Text** | Often includes letters for entropy |
| Phone number confirmation | **Number** | Standard phone-based SMS |
| API access tokens | **Text** | Typically alphanumeric for security |
| Payment authorization | **Password** | Sensitive - visual privacy needed |
| High-security scenarios | **Password** | Protection from shoulder-surfing |
| Time-based authenticator app | **Number** | Always numeric (6 digits) |
| Custom application code | **Text** | Maximum flexibility |

## Type-Specific Considerations

### Numeric (Number) Type
**Pros:**
- Matches SMS OTP standard (4-6 digits)
- Mobile keyboard shows numeric keypad
- Easier for users to read and enter

**Cons:**
- Limited character space
- Less entropy (fewer possible combinations)
- Vulnerable to brute force if timeout not implemented

**Best Practice:**
```csharp
// For SMS verification - implement timeout
var otpInput = new SfOtpInput
{
    Length = 6,
    Type = OtpInputType.Number
};

var expirationTime = DateTime.Now.AddMinutes(5);
otpInput.ValueChanged += async (s, e) =>
{
    if (DateTime.Now > expirationTime)
    {
        await DisplayAlert("Expired", "OTP code has expired", "OK");
        otpInput.IsEnabled = false;
    }
};
```

### Alphanumeric (Text) Type
**Pros:**
- Higher entropy (more possible combinations)
- Better for email-based codes
- More secure for sensitive operations

**Cons:**
- Case-sensitivity can cause user confusion
- Longer codes might be harder to enter manually
- Standard keyboard shown on mobile

**Best Practice:**
```csharp
// For email verification - consider case insensitivity
var otpInput = new SfOtpInput
{
    Length = 8,
    Type = OtpInputType.Text
};

otpInput.ValueChanged += (s, e) =>
{
    // Normalize to handle case-insensitive entry
    string normalized = e.NewValue?.ToUpper();
    // Then verify normalized version
};
```

### Password (Masked) Type
**Pros:**
- Visual privacy from observers
- Better for sensitive operations
- Reduces shoulder-surfing risk

**Cons:**
- User cannot verify entered characters visually
- May increase user anxiety
- Should be combined with other security measures

**Best Practice:**
```csharp
// For high-security scenarios - provide context
var stackLayout = new VerticalStackLayout
{
    Spacing = 10,
    Children =
    {
        new Label { Text = "Enter OTP from your authenticator app", FontSize = 14 },
        new Label { Text = "(Code is masked for your security)", FontSize = 12, TextColor = Colors.Gray },
        new SfOtpInput
        {
            Length = 6,
            Type = OtpInputType.Password
        }
    }
};
```

## Common Issues and Solutions

### Issue: User Cannot Enter Letters
**Problem:** Using `Type="Number"` but user needs to enter alphanumeric code
**Solution:** Change to `Type="Text"` to allow both letters and numbers

### Issue: Text Input Shows Wrong Keyboard
**Problem:** Mobile keyboard not optimized for input type
**Solution:** Ensure `Type` property matches expected input (Number = numeric keypad, Text = standard keyboard)

### Issue: Password Type Characters Not Hidden
**Problem:** Masked characters still visible
**Solution:** This behavior is platform-specific; ensure Syncfusion toolkit is properly registered in MauiProgram.cs
